Setup Guide

1 Quick Start

Get Poly running in your CI pipeline in under 2 minutes. Sign up, add the GitHub Action, and start seeing trust scores.

1

Sign up and get your API key

Create an account at your Poly instance. You'll receive an API key to authenticate requests.

2

Add the GitHub Action to your workflow

Add this step after your test runner. Make sure your tests output JUnit XML format.

YAML 
# .github/workflows/ci.yml
name: CI with Poly
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test -- --reporter junit --reporter-options output=test-results.xml
      - uses: Pritahi/poly-action@v1
        with:
          api_key: ${{ secrets.POLY_API_KEY }}
          junit_path: 'test-results/*.xml'

2 Python SDK

Use the Python SDK to ingest test results programmatically, query dashboards, and configure alerts.

Shell 
pip install poly-trust
Python 
from poly import Poly

# Initialize with your API key
p = Poly("your-api-key")

# Ingest a JUnit XML file
result = p.ingest_junit("path/to/junit.xml")
print(result["avg_trust_score"])

# Get dashboard data
dashboard = p.dashboard("my-repo")

# List all tests with trust scores
tests = p.list_tests("my-repo")

# Get details for a specific test
detail = p.test_detail("my-repo", "TestClass.test_method")

# View quarantined tests
quarantined = p.quarantined("my-repo")

# Configure alerts (Discord/Slack)
p.configure_alerts(
    repo_name="my-repo",
    webhook_url="https://discord.com/api/webhooks/...",
    channel_type="discord",
    min_trust_drop=20,
    alert_on_flaky=True,
    alert_on_quarantine=True
)

# Get badge URL for your README
url = p.badge_url("my-repo")

Available Methods

MethodDescription
ingest_junit(path)Ingest a JUnit XML file for a repo
ingest_results(results)Ingest raw test results (list of dicts)
dashboard(repo)Get aggregated dashboard data
list_tests(repo)List all tests with trust scores
test_detail(repo, name)Get detailed info for a single test
quarantined(repo)Get quarantined tests below threshold
configure_alerts(...)Set up Discord/Slack webhook alerts
badge_url(repo)Get the trust badge SVG URL

3 REST API Reference

All endpoints require the X-Poly-API-Key header for write operations. Read operations (GET) are open by default.

MethodPathDescription
POST /api/junit Ingest JUnit XML content. Body: {"xml_content": "...", "repo_name": "..."}
POST /api/runs Create a test run from raw results. Body: {"test_results": [...], "repo_name": "..."}
GET /api/dashboard?repo_name=X Get aggregated stats, flaky tests, recent runs, and trust distribution
GET /api/tests?repo_name=X List all tests with trust scores, pass rates, and flaky categories
GET /api/tests/{name}?repo_name=X Get detailed test info including history, score breakdown, and error patterns
GET /api/quarantined?repo_name=X Get tests below the trust threshold (default: 30)
GET /api/runs?repo_name=X Get recent CI runs for a repository (max 100)
GET /api/repos List all tracked repositories with aggregate stats
GET /badge/{repo_name} Get an SVG trust score badge for a repository
POST /api/alerts/config?repo_name=X Configure webhook alerts (Discord/Slack)
DELETE /api/tests/{name}?repo_name=X Delete all history for a specific test

Example Requests

cURL — Ingest JUnit 
curl -X POST https://your-poly-instance.com/api/junit \
  -H "Content-Type: application/json" \
  -H "X-Poly-API-Key: your-key" \
  -d '{"xml_content": "...", "repo_name": "my-repo", "branch": "main"}'

curl https://your-poly-instance.com/api/dashboard?repo_name=my-repo
cURL — Configure Alerts 
curl -X POST https://your-poly-instance.com/api/alerts/config?repo_name=my-repo \

  -H "X-Poly-API-Key: your-key" \
  -d '{"webhook_url": "https://discord.com/...", "channel_type": "discord"}'

4 Trust Score Algorithm

Each test receives a trust score from 0 to 100. The score is calculated from four weighted components plus an optional flaky penalty.

60% weight

Pass Rate

The percentage of recent runs that passed. A consistently passing test earns close to the full 60 points.

20% weight

Recent Trend

Compares pass rate of the last 5 runs vs. the previous 5. Improving tests get a bonus; degrading tests lose points.

10% weight

Duration Variance

Measures coefficient of variation in execution time. Consistent durations suggest stability; erratic timing indicates flakiness.

10% weight

Error Pattern Consistency

When a test fails, consistent error messages indicate a real bug, while varied errors suggest flakiness.

-5 to -20 penalty

Flaky Category Penalty

If a test is classified as flaky, an additional penalty is applied based on the category severity:

timing −5   order_dependency −10   shared_state −15   non_deterministic_data −20

Flaky Categories

Timing

High duration variance. Test execution time is unpredictable, suggesting race conditions or timeout sensitivity.

Order Dependency

Test outcome changes based on execution order. Frequent pass/fail transitions indicate isolation issues.

Shared State

Test shares mutable state with other tests. Fails intermittently with consistent error messages.

Non-deterministic Data

Test fails due to unpredictable external data, random values, or varying API responses.
