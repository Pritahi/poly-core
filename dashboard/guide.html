§1 Arraignment — Quick Start

Get Falsky running in your CI pipeline in under 2 minutes. Sign up, add the GitHub Action, and start seeing trust scores.

1

File Your Motion — Sign Up & Get API Key

Create an account at your Falsky instance. You'll receive an API key to authenticate requests — your credentials to enter the courtroom.

2

Present Your Evidence — Add the GitHub Action

Add this step after your test runner. Make sure your tests output JUnit XML format — the court requires standardized evidence.

YAML — Filing Procedure 
# .github/workflows/ci.yml
name: CI with Falsky
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test -- --reporter junit --reporter-options output=test-results.xml
      - uses: Pritahi/falsky-action@v1
        with:
          api_key: ${{ secrets.FALSKY_API_KEY }}
          junit_path: 'test-results/*.xml'

§2 Python SDK — Court Filing Interface

Use the Python SDK to ingest test results programmatically, query dashboards, and configure alerts — the official court filing system.

Shell — Court Clerk Installation 
pip install falsky-trust
Sworn & Subscribed
Python — Official Testimony 
from falsky import Falsky

# Initialize with your API key
p = Falsky("your-api-key")

# Ingest a JUnit XML file
result = p.ingest_junit("path/to/junit.xml")
print(result["avg_trust_score"])

# Get dashboard data
dashboard = p.dashboard("my-repo")

# List all tests with trust scores
tests = p.list_tests("my-repo")

# Get details for a specific test
detail = p.test_detail("my-repo", "TestClass.test_method")

# View quarantined tests
quarantined = p.quarantined("my-repo")

# Configure alerts (Discord/Slack)
p.configure_alerts(
    repo_name="my-repo",
    webhook_url="https://discord.com/api/webhooks/...",
    channel_type="discord",
    min_trust_drop=20,
    alert_on_flaky=True,
    alert_on_quarantine=True
)

# Get badge URL for your README
url = p.badge_url("my-repo")

§3 REST API — Court Records Reference

All endpoints require the X-Falsky-API-Key header for write operations. Read operations (GET) are open by default — public record.

MethodPathDescription
POST /api/junit Ingest JUnit XML content. Body: {"xml_content": "...", "repo_name": "..."}
POST /api/runs Create a test run from raw results. Body: {"test_results": [...], "repo_name": "..."}
GET /api/dashboard?repo_name=X Get aggregated stats, flaky tests, recent runs, and trust distribution
GET /api/tests?repo_name=X List all tests with trust scores, pass rates, and flaky categories
GET /api/tests/{name}?repo_name=X Get detailed test info including history, score breakdown, and error patterns
GET /api/quarantined?repo_name=X Get tests below the trust threshold (default: 30)
GET /api/runs?repo_name=X Get recent CI runs for a repository (max 100)
GET /api/repos List all tracked repositories with aggregate stats
GET /badge/{repo_name} Get an SVG trust score badge for a repository
POST /api/alerts/config?repo_name=X Configure webhook alerts (Discord/Slack)
DELETE /api/tests/{name}?repo_name=X Delete all history for a specific test

§4 Trust Score — Sentencing Guidelines

Each test receives a trust score from 0 to 100. The score is calculated from four weighted components plus an optional flaky penalty — the court's sentencing matrix.

Exhibit A — 60% weight

Pass Rate

The percentage of recent runs that passed. A consistently passing test earns close to the full 60 points — the primary evidence of reliability.

Exhibit B — 20% weight

Recent Trend

Compares pass rate of the last 5 runs vs. the previous 5. Improving tests get a bonus; degrading tests lose points — the trajectory of conduct.

Exhibit C — 10% weight

Duration Variance

Measures coefficient of variation in execution time. Consistent durations suggest stability; erratic timing indicates flakiness — behavioral consistency under oath.

Exhibit D — 10% weight

Error Pattern Consistency

When a test fails, consistent error messages indicate a real bug, while varied errors suggest flakiness — credibility of testimony.

Penalty: −5 to −20

Flaky Category — Aggravating Circumstances

If a test is classified as flaky, an additional penalty is applied based on the category severity — the court's enhanced sentencing:

timing −5   order_dependency −10   shared_state −15   non_deterministic_data −20

Flaky Categories — Criminal Classifications

Timing

High duration variance. Test execution time is unpredictable, suggesting race conditions or timeout sensitivity.

Order Dependency

Test outcome changes based on execution order. Frequent pass/fail transitions indicate isolation issues.

Shared State

Test shares mutable state with other tests. Fails intermittently with consistent error messages.

Non-deterministic Data

Test fails due to unpredictable external data, random values, or varying API responses.
