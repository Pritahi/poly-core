§1
Arraignment — Quick Start
Get Poly running in your CI pipeline in under 2 minutes. Sign up, add the GitHub Action, start receiving trust scores.
1

File Your Motion — Sign Up & Get API Key

Create an account at your Poly instance. You'll receive an API key to authenticate requests — your pass to enter the station.

2

Present Evidence — Add GitHub Action

Add this step after your test runner. Tests must output JUnit XML format — the station requires standardized evidence.

YAML — Filing Procedure
# .github/workflows/ci.yml
name: CI with Poly
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test -- --reporter junit --reporter-options output=results.xml
      - uses: Pritahi/falsky-action@v1
        with:
          api_key: ${{ secrets.POLY_API_KEY }}
          junit_path: 'test-results/*.xml'

§4
Trust Score Algorithm — Sentencing Matrix
Each test receives a 0-100 trust score from four weighted components plus optional flaky penalty.
Exhibit A — 60%

Pass Rate

The percentage of recent runs that passed. Primary reliability signal.

Exhibit B — 20%

Recent Trend

Last 5 runs vs previous 5. Improving = bonus, degrading = penalty.

Exhibit C — 10%

Duration Stability

Coefficient of variation in execution time. Consistent = stable.

Exhibit D — 10%

Error Consistency

Consistent errors = real bug. Varied errors = flaky witness.

Aggravating Circumstances

Flaky Category Penalty

If classified flaky, additional penalty applied by severity:

timing −5    order_dependency −10    shared_state −15    non_deterministic_data −20

Timing

High duration variance — suggests race conditions or timeouts.

Order Dependency

Outcome changes by execution order — isolation issues suspected.

Shared State

Mutable shared state causes intermittent failures.

Non-Deterministic Data

Unpredictable external inputs cause sporadic failures.
